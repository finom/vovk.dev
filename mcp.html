MCP Server

MCP Server

Now that we’ve covered Function Calling, Real-time UI, and Real-Time Database Polling, it’s time to add the final piece of the puzzle and demonstrate that your API is an MCP when built with Vovk.ts. To achieve this, we’ll use the mcp-handler  package, which enables us to easily create MCP servers on Next.js.

The createLLMTools function accepts a resultFormatter option that takes a function to format the tool execution result before it’s returned to the LLM. It also accepts a special string value mcp that formats the result to match MCP server expectations. This is all we need to make our tools output MCP-compatible and return results in the following format:

{
    content: [
      {
        type: 'text',
        text: 'Result text',
      },
    ],
}
import { createLLMTools } from "vovk";
import UserController from "@/modules/user/UserController";
import TaskController from "@/modules/task/TaskController";
 
const { tools } = createLLMTools({
  modules: {
    UserController,
    TaskController,
  },
  resultFormatter: "mcp",
});

This is where custom operation properties such as x-tool-successMessage and x-tool-errorMessage become useful for making the output more LLM-friendly.

import { prefix, get, put, post, del, operation } from "vovk";
import UserService from "./UserService";
import { z } from "zod";
import { BASE_FIELDS } from "@/constants";
import { UserSchema } from "../../../prisma/generated/schemas";
import { withZod } from "@/lib/withZod";
 
@prefix("users")
export default class UserController {
  @operation({
    summary: "Update user",
    description:
      "Updates an existing user with the provided details, such as their email or name.",
    "x-tool-successMessage": "User updated successfully",
    "x-tool-errorMessage": "Failed to update user",
  })
  @put("{id}")
  static updateUser = withZod({
    body: UserSchema.omit(BASE_FIELDS).partial(),
    params: UserSchema.pick({ id: true }),
    handle: async ({ vovk }) =>
      UserService.updateUser(vovk.params().id, await vovk.body()),
  });
 
  // ...
}

This will set the text property of the output to “User updated successfully” or “Failed to update user” depending on the operation result, concatenated with the actual serialized result or error message.

{
    content: [
      {
        type: 'text',
        text: 'User updated successfully\nResult: { id: 1, email: "user@example.com" }',
      },
    ],
}

Now, create the route file according to the used MCP library documentation .

We’ll use UserController and TaskController as the source of the MCP tools. The controller methods will be executed in the same context rather than making HTTP requests, as they follow the rules of callable handlers.

route.ts
1import { createMcpHandler } from "mcp-handler";
2import { createLLMTools } from "vovk";
3import UserController from "@/modules/user/UserController";
4import TaskController from "@/modules/task/TaskController";
5import { convertJsonSchemaToZod } from "zod-from-json-schema";
6import { mapValues } from "lodash";
7
8const { tools } = createLLMTools({
9  modules: {
10    UserController,
11    TaskController,
12  },
13  resultFormatter: "mcp",
14  onExecute: (result, { moduleName, handlerName, body, query, params }) =>
15    console.log(`${moduleName}.${handlerName} executed`, {
16      body,
17      query,
18      params,
19      result,
20    }),
21  onError: (e) => console.error("Error", e),
22});
23
24const handler = createMcpHandler(
25  (server) => {
26    tools.forEach(({ name, execute, description, parameters }) => {
27      server.tool(
28        name,
29        description,
30        mapValues(parameters?.properties ?? {}, convertJsonSchemaToZod),
31        execute,
32      );
33    });
34  },
35  {},
36  { basePath: "/api" },
37);
38
39export { handler as GET, handler as POST };

The parameters argument passed to the server.tool method expects Zod schemas, so we need to convert the JSON Schema parameters of each tool to Zod schemas using the zod-from-json-schema  package. This approach works with all available validation libraries. However, if you’re already using Zod to validate incoming requests, you can use the models property of a tool to get Zod schemas directly, as shown below:

// ...
const handler = createMcpHandler(
  (server) => {
    tools.forEach(({ name, execute, description, models }) => {
      server.tool(
        name,
        description,
        models,
        execute,
      );
    });
  },
  {},
  { basePath: "/api" },
);
// ...

Note that the models property is only available for callable handlers and not for RPC modules.

As a reminder, you can also use third-party OpenAPI Mixins to add more tools to your MCP server.

import { GithubIssuesRPC } from "vovk-client";
// ...
const githubOptions = {
    init: {
        headers: {
            Authorization: `Bearer ${process.env.GITHUB_TOKEN}`,
            "X-GitHub-Api-Version": "2022-11-28",
        },
    },
};
const { tools } = createLLMTools({
    modules: {
        UserController,
        TaskController,
        GithubIssuesRPC: [GithubIssuesRPC, githubOptions],
    },
});
// ...
